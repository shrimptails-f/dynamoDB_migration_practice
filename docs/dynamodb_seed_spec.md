# DynamoDB Seed / Migration 方針

## 基本方針

ローカル環境における初期データ投入は seed として扱い、物理定義の変更に伴う既存データの補完や移行は migration として扱う。

- seed は開発・検証の再現性確保を目的とする
- migration は既存データの整合性維持を目的とする
- seed と migration は責務を分けて管理する

## Seed の位置づけ

ローカル環境では `destroy & deploy` を前提とするため、テーブル再作成後に seed を投入できる状態を維持する。

- seed は `tools/seeder` に実装する
- seed はローカル開発時の動作確認に必要な初期データを投入する
- seed は何度実行しても破綻しないよう、再実行可能な実装にする
- seed は本番データ移行の用途では使用しない

## Seed の実装方針

- DynamoDB への接続には `internal/library/dynamodb/client.go` を利用する
- 実行環境の判定には既存の環境変数を利用する
- データは固定値または決定的に生成される値を優先する
- 同一 PK / SK のレコードは上書き可能な前提で投入する

## Migration の位置づけ

以下のような変更では、IaC の更新だけでなく個別 migration を検討する。

- GSI 追加後に既存データのバックフィルが必要な場合
- 属性名変更により旧形式と新形式が混在する場合
- 既存データへ新属性を補完する必要がある場合
- PK / SK の見直しが必要な場合

seed は初期データ投入のための仕組みであり、migration の代替にはしない。

## Migration の実施方針

- migration は必要なタイミングで個別実装する
- 本番向け migration は seed とは別の実装として管理する
- migration は再実行可能性を考慮して実装する
- 実行時は対象環境、対象件数、実行ログを確認できるようにする
- 大規模変更ではバックアップ、切り替え手順、ロールバック手順を事前に定義する

## 将来的な運用

seed の対象データや件数が増えた場合は、用途ごとにデータ定義を分離することを検討する。

- 例: `local` 用 seed
- 例: `integration test` 用 seed

migration が増えて定型化してきた場合は、専用ディレクトリや運用ジョブの整備を検討する。
