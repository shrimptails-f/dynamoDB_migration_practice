# DynamoDB 管理方針

## 基本方針

ローカル環境・本番環境を問わず、DynamoDB の物理定義は IaC (CDK/Terraform) で管理する。

- テーブル
- PK / SK
- GSI
- TTL
- Billing mode
- そのほかテーブル設定

ローカルでエミュレーターを使う場合でも IaC で管理する。

- 本番との差分を減らせる
- 誰が立ち上げても同じテーブル定義を再現できる
- GSI や TTL を含めた構成をコードとして管理できる

## 開発環境

開発環境では、テーブル定義の変更があれば `destroy & deploy` で作り直す運用を基本とする。

- `develop` ブランチが常に最新のテーブル定義
- データ保持よりも再現性を優先する

## 本番環境

本番環境では PK / SK は不変の前提とする。  
※変更が必要になった場合は新テーブルまたは新キー系を並行導入する

- additive な変更は IaC で管理する
- アプリケーションは後方互換を維持しながら段階的に変更する

ここでいう additive な変更の例:

- GSI の追加
- GSI の削除
- TTL の有効化
- 非キー属性の追加

## マイグレーションの考え方

DynamoDB はスキーマレスのため、RDB のような厳密な migration 機構は必須ではない。

一方で、以下のような変更では migration 的な対応が必要になる。

- GSI の追加後に既存データの整合を取りたい場合
- 属性名変更で旧形式と新形式が混在する場合
- 既存データにバックフィルが必要な場合
- PK / SK の見直しが必要な場合

したがって、方針としては以下とする。

- 専用の migration 基盤は必須ではない
- ただし、データ移行やバックフィルが必要な変更では個別対応を行う

## GSI 変更時の注意点

| 操作 | 本番への影響 |
|------|------------|
| GSI 追加 | 無停止で可能 ✅ |
| GSI 削除 | 無停止で可能 ✅ |
| GSI の Key 変更 | 削除→再作成 (一時的にクエリ不可) ⚠️ |

本番で GSI の Key を変えたい場合は以下の手順が必要。

1. 古い GSI を残しつつ新しい GSI を追加
2. 新しい GSI へバックフィル
3. アプリを新しい GSI に切り替え
4. 旧 GSI を削除

## 大規模変更の扱い

PK / SK の変更や大規模なデータ構造変更は、通常の IaC 更新だけでは扱わない。

以下を都度検討する。

- バックアップ取得
- 新旧テーブルまたは新旧 GSI の並行運用
- バックフィル
- 切り替え手順
- ロールバック手順

## 役割分担

| 役割 | 内容 |
|------|------|
| IaC | テーブル、PK/SK、GSI、TTL などの物理定義を管理する |
| アプリコード | additive な属性変更を後方互換を保ちながら吸収する |
| 運用ジョブ / スクリプト | バックフィルや個別のデータ移行を行う |
